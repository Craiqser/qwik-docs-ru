# Маршрутизация

Маршрутизация - это способ сопоставления публичных URL-адресов сайта с определёнными компонентами, объявленными в вашем приложении.

Qwik City использует маршрутизацию на основе каталогов. Это означает, что структура вашего каталога routes определяет URL-адреса, которые пользователь сможет увидеть в приложении. Однако, она несколько отличается от традиционной маршрутизации на основе файлов, о чём мы и поговорим.

## Маршрутизация на основе каталогов

Ниже описываются типы маршрутизации, поддерживаемые Qwik City.

В каталоге routes, `путь к каталогу` плюс файл `index` представляют собой URL-адрес. Например, если пользователь видит `https://example.com/some/path`, то будет показан компонент, экспортированный по адресу `src/routes/some/path/index` (расширение файла может быть, например, `.mdx`, или `.tsx`).

```
src/
└── routes/
    └── some/
        └── path/
            └── index.tsx       # https://example.com/some/path
```

Обратите внимание, что конечный файл маршрута имеет название index. Это **важно**. В других мета-фреймворках, с которыми вы можете быть знакомы, существует различие между `веб-страницами` и `конечными точками`, работающими с данными, или `API-маршрутами`. В Qwik City все они являются `конечными точками`.

Чтобы определить конечную точку, вы должны создать файл `index.[расширение]`, где `[расширение]` может быть одним из `.ts`, `.tsx`, `.js`, `.jsx`, `.md` или `.mdx`.

Следующая структура каталогов

```
src/
└── routes/
    ├── contact.tsx
    ├── about.tsx
    ├── store.tsx
    └── index.tsx
```

может работать в других мета-фреймворках, использующих маршрутизацию на основе файлов, но Qwik City не будет регистрировать маршрут для других файлов, кроме индексного. В Qwik City эквивалентная структура будет выглядеть следующим образом:

```
src/
└── routes/
    ├── contact/
    │   └── index.tsx          # https://example.com/contact
    ├── about/
    │   └── index.tsx          # https://example.com/about
    ├── store/
    │   └── index.tsx          # https://example.com/store
    └── index/
        └── index.tsx          # https://example.com/index
```

Сначала это может показаться излишним, но у такого подхода есть свои преимущества. Одним из преимуществ является возможность определять файлы компонентов в каталоге маршрута без их отрисовки. Рассмотрим следующую структуру:

```
src/
└── routes/
    ├── index.tsx                     # https://example.com/
    └── some/
        ├── index.tsx                 # https://example.com/some
        └── path/
            ├── index.tsx             # https://example.com/some/path
            └── other_component.tsx   # файл игнорируется, поскольку он не является индексным.
```

Файл `other_component.tsx` может быть импортирован и использован внутри `path/index.tsx`, иначе он просто игнорируется Qwik City.

## Реализация компонента

Чтобы вернуть HTML для определённого маршрута, необходимо реализовать компонент. Для файлов `.tsx` компонент должен быть экспортирован по умолчанию (`export default`). В качестве альтернативы можно использовать расширение `.mdx`, рассмотренное [далее](../../../qwikcity/content/mdx/index.mdx).

```tsx
export default component$(() => {
  return <H1>Hello World!</H1>;
});
```

## `@qwik-city-plan`

Qwik City хранит информацию о маршрутах в файлах на диске. Это отлично подходит для эргономики разработчика, но не подходит для создания бандлов и чанков. Кроме того, в некоторых средах, таких как облачные функции, нет файловой системы, к которой можно получить доступ. По этой причине необходимо сериализовать информацию о маршруте в один файл. Это делается с помощью импорта `@qwik-city-plan`.

```tsx
import cityPlan from '@qwik-city-plan';
```

Импорт `@qwik-city-plan` является синтетическим, то есть он создается в процессе сборки. Он содержит всю информацию из каталога `/src/routes` в формате JavaScript. Синтетический импорт обеспечивается плагином vite `qwikCity()`, который доступен из `@builder.io/qwik-city/vite`.
