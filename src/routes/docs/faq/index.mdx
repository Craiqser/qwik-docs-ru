# FAQ

## *@QwikDev* - разумный ИИ?

Да, а ещё я весёлый! [Подпишись на меня](https://twitter.com/QwikDev)

## Почему он называется Qwik?

Сначала он назывался *qoot*, но мы решили, что его будет трудно найти. Один из наших друзей, [@patrickjs__](https://twitter.com/PatrickJS__), придумал Qwik, и, после внутреннего голосования на [builder.io](https://www.builder.io/), мы изменили название!

## Зачем еще один фреймворк?

Если кратко, то Qwik решает проблему, которую не могут решить другие фреймворки. Qwik обеспечивает мгновенный запуск независимо от сложности приложения. То есть, приложения Qwik предоставляют одинаковое количество начального JS независимо от количества компонентов. [Qwik - первый О(1)-фреймворк с открытым исходным кодом](https://www.builder.io/blog/our-current-frameworks-are-on-we-need-o1).

## Что такое Qwik City?

[Qwik City](./../../qwikcity/overview/index.mdx) - это дополнительный набор API поверх Qwik. Считайте, что *Qwik* - это ядро, а *City* - дополнительные возможности (маршрутизация, загрузка данных, конечные точки...).

## Сложен ли Qwik в освоении?

Мы разработали Qwik таким образом, что он [чрезвычайно прост в освоении](./../cheat/qwik-react/index.mdx) и знаком React-разработчикам, создание компонентов практически идентично, а маршрутизация вдохновлена Nextjs и другими фреймворками.

Однако есть и принципиально [новые концепции](./../think-qwik/index.mdx), такие как [возобновляемость](./../concepts/resumable/index.mdx), мелкозернистая реактивность, но мы думаем, что кривая обучения не будет крутой.

У нас также есть интерактивный [учебник](./../../tutorial/welcome/overview/).

## Что значат все эти "$"?

Вы могли заметить, что в приложениях Qwik больше [`$`](./../advanced/optimizer/index.mdx#the-meaning-of-), чем обычно: `component$()`, `useWatch$()`, `<div onClick$={() => console.log('click')} />`. Это не просто так. Qwik разбивает ваше приложение на небольшие фрагменты, меньшие, чем сам компонент: обработчики событий, хуки и прочие. Знак `$` сигнализирует как [оптимизатору](./../advanced/optimizer/index.mdx), так и разработчику, когда это происходит.

**Пример:**

```tsx
import { component$ } from '@builder.io/qwik';

export const App = component$(() => {
  console.log('render');
  return <p onClick$={() => console.log('hello')}>Hello Qwik</p>;
});
```

Компонент выше разделен благодаря `$` на:

**app.js**
```js
import { componentQrl, qrl } from "@builder.io/qwik";

const App = /*#__PURE__*/ componentQrl(qrl(()=>import('./app_component_akbu84a8zes.js'), "App_component_AkbU84a8zes"));

export { App };
```

**app_component_akbu84a8zes.js**

```js
import { jsx as _jsx } from "@builder.io/qwik/jsx-runtime";
import { qrl } from "@builder.io/qwik";
export const App_component_AkbU84a8zes = ()=>{
    console.log('render');
    return /*#__PURE__*/ _jsx("p", {
        onClick$: qrl(()=>import("./app_component_p_onclick_01pegc10cpw"), "App_component_p_onClick_01pEgC10cpw"),
        children: "Hello Qwik"
    });
};
```

**app_component_p_onclick_01pegc10cpw.js**

```js
export const App_component_p_onClick_01pEgC10cpw = ()=>console.log('hello');
```

## Qwik загружает JS при взаимодействии с пользователем?

Нет, в рабочей среде Qwik использует много информации, сгенерированной во время SSR, чтобы как можно скорее (ASAP) начать [предварительную загрузку](./../../qwikcity/prefetching/overview/index.mdx) фрагментов интерактивности только для текущей страницы. Таким образом, к моменту взаимодействия с пользователем, JS уже загружен.

## Если Qwik предварительно загружает JS, то в чём разница?

Предварительная загрузка - это не то же самое, что парсинг и выполнение JS, Qwik не выполняет JS, пока пользователь не взаимодействует с ним.

Кроме того, [предварительная загрузка](./../../qwikcity/prefetching/overview/index.mdx) позволяет Qwik расставить приоритеты от более важных фрагментов интерактивности к наименее важным.

Например, кнопка "Купить сейчас" важнее, чем кнопка "Добавить в корзину", поэтому Qwik сначала загрузит кнопку "Купить сейчас", а затем кнопку "Добавить в корзину".

Qwik не нуждается в полной предварительной загрузке, чтобы начать работу, в то время как другим фреймворкам из-за [гидратации](https://www.builder.io/blog/hydration-is-pure-overhead) необходимо загрузить целиком критически-важные данные, прежде чем они смогут продолжить работу.

## В низкоскоростных сетях приложения Qwik тоже медленны?

Вовсе нет, благодаря [предварительной загрузке](./../../qwikcity/prefetching/overview/index.mdx) приложения Qwik страдают от низкоскоростных сетей не больше, чем на других фреймворках. Фактически, благодаря точечной фрагментации и возобновляемости, приложения Qwik могут стать интерактивными с гораздо меньшим количеством JS, что эффективно ускоряет их работу в медленных сетях.

## Qwik генерирует слишком много маленьких файлов?

В среде разработки Qwik генерирует много маленьких файлов, поскольку использует сервер разработки [Vite.js](https://vitejs.dev/), но в рабочей среде Qwik эффективно объединяет файлы.

## Почему Qwik использует JSX? У него под капотом React?

Нет, React не используется вообще, но Qwik использует JSX в качестве синтаксиса шаблонов.

JSX - это не React, на самом деле JSX - это просто синтаксис без конкретной семантики. Мы выбрали JSX по нескольким причинам:

- **Очень простой синтаксис**: не изобретает колесо, а использует существующий JS для циклов, условий и т.д., [спецификация JSX удивительно проста и мала](https://facebook.github.io/jsx/)
- **Экосистема**: он хорошо поддерживается IDE, линтерами, инструментами аудита безопасности, отладки, подсветки...
- **Похож на HTML**: JSX визуально и концептуально похож на HTML-дерево. Другие системы шаблонов, такие как *html templates* (lit-html), представляют собой не деревья, а массивы лексем, что усложняет преобразование и работу с ними.
- **Популярен**: Нравится вам это или нет, но JSX является наиболее широко используемым синтаксисом шаблонов в мире.

## Существует ли маршрутизатор для Qwik?

ДА! [Qwik City](./../../qwikcity/overview/index.mdx) включает в себя маршрутизатор на основе каталогов, и он вдохновлен Nextjs и другими фреймворками.

## Для развёртывания приложений Qwik нужен сервер?

Вы легко можете развернуть приложение Qwik в любой [бессерверной среде благодаря нашим адаптерам](./../../qwikcity/adaptors/overview/index.mdx), мы также поддерживаем адаптер [Node](./../../qwikcity/adaptors/node/index.mdx) для серверов на базе Nodejs, таких, например, как Express.

Вы можете развернуть своё приложение Qwik как статический сайт, если нет необходимости в SSR, благодаря нашему [SSG-адаптеру](./../../qwikcity/static-site-generation/overview/index.mdx).

## SPA быстрее, чем MPA?

Зависит от ситуации, для SPA большая часть стоимости оплачивается заранее, за счёт полной загрузки в начале сессии. Таким образом, когда пользователь взаимодействует с приложением, затраты минимальны.

## MPA быстрее, чем SPA?

Зависит от ситуации. MPA загружаются чрезвычайно быстро, потому что им не нужно загружать много JS, но, когда пользователь выполняет навигацию, обычно происходит полная перезагрузка страницы.

Полная перезагрузка страницы обычно выполняется очень быстро, поскольку браузеры чрезвычайно быстро загружают и парсят HTML, но подход MPA не для всех, поскольку иногда хочется сохранить состояние между переходами, и SPA делает это очень хорошо.

Qwik - это уникальный фреймворк, который является одновременно и MPA и SPA.

## Можно ли на Qwik создать SPA?

Конечно! [Qwik City](./../../qwikcity/overview/index.mdx) включает компонент `<Link>`, который активирует SPA-навигацию.
С Qwik разработчикам не нужно выбирать между SPA и MPA, каждый сайт - это и то, и другое одновременно.

MPA или SPA - это уже не архитектурное решение, принимаемое в начале проекта, а решение, принимаемое для каждой ссылки.

## Может ли Qwik генерировать статические сайты (SSG)?

Да! Эта возможность есть во всех стартовых шаблонах Qwik City. Узнать, как сделать [Static Site Generation](./../../qwikcity/static-site-generation/overview/index.mdx).

## Но... с другими фреймворками я также могу делать MPA и SPA

Не совсем так, другие фреймворки предлагают удалить все `<Scripts>` в корне для создания MPA, фактически удаляя всю интерактивность вместе с навигацией SPA.

А если скрипты _не_ удаляются, то каждая перезагрузка страницы становится очень дорогой, потому что каждая перезагрузка страницы означает, что фреймворк должен гидратировать всю страницу. А вот Qwik не имеет [стоимости гидратации](https://www.builder.io/blog/hydration-is-pure-overhead) для каждой загрузки страницы.

## Будет ли миграция на Qwik трудоемкой?

Не обязательно. Если вы пришли из React, то перенос ваших компонентов в Qwik будет довольно прост. Помимо этого, благодаря `Qwik React` вы можете использовать всю экосистему React, вы можете использовать любые ваши React-компоненты и любые React-библиотеки в приложении Qwik.

## Могу ли я пользоваться экосистемой React?

ДА! Qwik может нативно запускать React-компоненты с помощью библиотеки `@builder.io/qwik-react`, добавьте её в ваше приложение:

```
npm run qwik add qwik-react
```

Вы будете поражены!

## Qwik делает частичную гидратацию?

Нет, частичная гидратация (или островная архитектура), популяризированная [Astro](https://astro.build/), предназначена для предотвращения [гидратации всей страницы](https://www.builder.io/blog/hydration-is-pure-overhead), когда все компоненты страницы должны быть загружены и выполнены, и вместо этого приложение разбивается на островки интерактивности.

Острова, которые разработчикам необходимо определять вручную, а затем вручную описывать, в каких ситуациях должна происходить их гидратация. Острова, которые не могут общаться друг с другом.

В противовес этому, компоненты Qwik вообще не гидратируются. Qwik достигает этого с помощью мощной системы сериализации, которая сериализует только необходимое состояние графа реактивности, поэтому приложение может начать работу без предварительного запуска какого-либо JS.

Мы думаем, что возобновляемость масштабируется без негативных последствий частичной гидратации.

ДА, существует растущее сообщество разработчиков Qwik в [Discord](https://qwik.builder.io/chat) и [Github](https://github.com/BuilderIO/qwik), они вносят удивительный вклад в развитие фреймворка, создают масштабные сайты и помогают друг другу. [Присоединяйтесь](https://qwik.builder.io/chat).

## Готов ли Qwik к запуску в рабочей среде?

ДА! Но это зависит от обстоятельств. Qwik находится в разработке уже 2 года, и мы достигли [вехи бета-версии](https://www.builder.io/blog/qwik-and-qwik-city-have-reached-beta), что означает нашу уверенность в готовности к запуску в рабочей среде, и не ожидается никаких ломающих изменений.

[Builder.io](https://www.builder.io/) и многие другие команды уже используют Qwik в рабочей среде, так что вы будете не одиноки.

## Полная перезагрузка страницы медленнее, чем SPA?

Зависит от ситуации. Полная перезагрузка страницы выполняется чрезвычайно быстро, поскольку браузеры чрезвычайно быстро загружают и парсят HTML, но подход MPA не для всех, поскольку иногда хочется сохранить состояние между переходами, и SPA делает это очень хорошо.

MPA может быть медленным в существующих фреймворках при попытке сделать MPA с интерактивностью, потому что существующие фреймворки должны будут запускать [гидратацию во время загрузки, что может быть очень медленным](https://www.builder.io/blog/hydration-is-pure-overhead).

## Qwik сериализует слишком много данных в HTML?

Нет, Qwik сериализует только те данные, которые необходимы для текущей страницы. Если на странице 1000 компонентов, но только один из них является интерактивным, то количество сериализованных данных будет пропорционально объёму интерактивности, а не количеству компонентов.
